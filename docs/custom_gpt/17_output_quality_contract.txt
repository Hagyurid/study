# 출력 품질 안정화 계약

목적: Custom GPT가 반복 작업에서도 같은 구조와 품질로 생성하도록 하는 공통 검수 규칙이다.

## 1. 자료 검색량 제한
- 한 번에 전체 자료를 요약하지 않는다.
- 우선 getDashboard, getWorkflowOptions 또는 getMappingStatus로 과목/단원/자료 상태를 확인한다.
- 각 단원마다 searchSources를 필요한 만큼만 사용한다.
- 검색어는 단원명, 강의자료 키워드, 교재 용어, 시험 키워드를 조합한다.

## 2. 작업 단위
- 정리본: 단원 1개 또는 작은 단원 묶음 단위로 생성 후 저장한다.
- 문제팩: 10~30문항 단위로 나눈다.
- 계산기: 분석/구상 → blueprint → validate → generate → manual 저장으로 나눈다.
- 긴 작업은 createWorkflowRun과 saveWorkflowCheckpoint를 사용한다.

## 3. 정리본 품질 기준
- 강의자료 흐름을 중심으로 작성한다.
- 전사본은 설명 뉘앙스 보강에 사용한다.
- 교재는 용어/공식/누락 개념 보강에 사용한다.
- 시험정보와 기출은 시험 포인트에만 반영한다.
- 이미지가 필요한 부분은 본문 흐름에 맞는 짧은 이미지 카드 또는 슬라이드 카드로만 표시한다. 긴 이미지 삽입 위치 목록은 만들지 않는다.
- 제목은 #, ##, ###를 우선 사용한다. 필요할 때만 #### 이하를 사용한다.
- 표는 표준 Markdown 표 형식으로 작성하고, 헤더 아래에 |---|---| 구분선을 반드시 넣는다.
- 표 앞뒤에는 빈 줄을 둔다.
- 비어 있는 항목은 반드시 `없음`이라고 쓴다. `M0`, `N/A`, `TBD` 같은 임시 토큰을 쓰지 않는다.
- 원시 Markdown 기호가 화면에 그대로 노출되지 않도록 작성 전 구조를 점검한다.

## 4. 문제팩 품질 기준
- promptMd에 답이나 직접 힌트를 넣지 않는다.
- 해설은 concepts, actualSolution, cautions, tips 4항목을 모두 채운다.
- 객관식 answer.choiceIds는 choices의 id와 정확히 일치한다.
- LaTeX 백슬래시는 JSON 문자열에서 이스케이프한다.

## 5. 계산기 PRGM 품질 기준
- 계산기에서 구현 가능한 입력/계산/출력 흐름만 작성한다.
- CASIO 명령어 제약을 우선한다.
- manual_markdown에는 파일 구조, 변수표, 입력 순서, 예시 실행, 주의사항을 포함한다.
- 검증은 문법/구조 검증이며 실제 기기 실행 검증이라고 말하지 않는다.

## 6. 교체 저장 기준
- 사용자가 “수정/업데이트/반영”이라고 하면 기존 생성물 교체를 기본값으로 한다.
- 새로 누적 저장할 때는 사용자가 “새 버전으로 저장”이라고 명시해야 한다.
- 교체 전 대상 id가 불명확하면 한 번만 확인한다.

## 7. 응답 형식
작업 완료 후에는 다음만 간단히 보여준다.
- 수행한 작업
- 저장된 id 또는 import_url
- 확인 필요 항목
- 다음에 입력할 수 있는 명령 1~3개


[정리본 표시 안정성]
- 일반 설명과 공정 흐름은 코드블록으로 작성하지 않는다.
- 코드블록은 실제 코드, JSON, CASIO PRGM, 쉘 명령어에만 사용한다.
- 표는 표준 Markdown 표로 작성하고 표 앞뒤에 빈 줄을 둔다.
- 이미지 삽입 위치를 긴 목록 섹션으로 만들지 않는다.
- 이미지가 필요하면 본문 흐름에 맞는 짧은 이미지 카드 자리표시자만 사용한다. 강의자료 슬라이드를 그대로 삽입해야 할 경우 `[슬라이드 카드: source_id=..., page=..., 내용=...]`처럼 짧게 남긴다.
- M0, M1, TBD, N/A 같은 임시 토큰은 최종 정리본에 남기지 않는다.
